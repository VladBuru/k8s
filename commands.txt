* ReplicaSet *
    Объект k8s, который следит и гарантирует,
    что в кластере всегда будут работать заданное количество одинаковых подов.
    - Label - пара ключ/значение которая вешается на объект k8s. Например на Pod
        label нужны для идентификации и группировки объектов. Много объектов могут иметь один и тот же label
    - Selector - правило, по которому отбираются объекты по их label.
        Он говорит k8s: найди объекты у которых есть такие label
    - Как это работает? ReplicaSet сканирует все поды и ищет определенные по label.
        Потом ReplicaSet считает количество подходящих подов с помощью Selector,
        если их меньше, чем нужно, то создает новые,
        а если больше - то удаляет лишние.
    Если попробовать изменить манифест replicaset (например, обновить версию контейнера),
    то через команду apply версия уже поднятых подов не поменяется (!)
    так как replicaset ТОЛЬКО следит за количеством поднятых реплик и ничем другим не занимается.
    Поменяется только при поднятии новых подов, так как манифест уже другой.

* Deployment *
    Объект k8s, который управляет развертываем и жизненным циклом приложения в контейнерах.
    Нужен, когда нам необходимо изменить приложение
    Он автоматизирует создание и поддержку указанных идентичных реплик (подов),
    бесперебойное обновление (постепенное обновление старых версий на новые, т.е. без остановки работы приложения),
    откат до предыдущих версий при ошибках, масштабирование (увеличение или уменьшение числа работающих экземпляров)
    Самое главное - он управляет ReplicaSet. Если ReplicaSet управляет конкретными подами
    с одним label (следит за тем чтобы было поднято необходимое количество реплик),
    то Deployment управляет уже ReplicaSet (отдает ему команды) и описывает правила поднятия, остановки и т.д.
    Параметр strategy: rollingUpdate - стратегия обновления приложения, при которой указывается
    maxSurge (максимальное число, на которое можно увеличить количество реплик при обновлении)
    и maxAvailable (максимальное число недоступных подов при обновлении)
    и происходит следующее - поднимается 1 новая реплика с обновлением, убивается одна старая,
    затем опять поднимается новая и убивается старая. И так пока все реплики не будут обновлены
    Теперь, когда манифест Deployment написан, можно поменять в нем версию,
    сделать apply и на всех подах поменяется версия без проблем.
    Также Deployment дает возможность ограничивать ресурсы подов.
        limits - сколько максимально ресурсов получит под, т.е. ограничитель.
            При превышении cpu происходит Throutling - замедление
            При превышении memory - под просто убивается
        requests - гарантированные ресурсы которые выделяются для пода.
        Шедулер который распределяет поды по нодам будет смотреть на параметры requests для распределения

* Probes *
    Проверки жизнеспособности, определяет, когда нужно перезапустить контейнеры.
    Liveness Probe
        Если проба проваливается, то k8s убивает контейнер и поднимает новый.
        Например сделали запрос, система легла, то зачем держать поднятым контейнер.
    Readines Probe
        Если Readines Probe провалилась, то контейнер не убивается и поднимается,
        а на него не пускаем трафик запросов
    Startup Probe
        Проба, которая отключает предыдущие пробы на момент поднятия приложения.
        То есть не пускает отправлять запросы с пробами другим пробам.

* Configmap *
    Предназначен для хранения конфигурационных данных в виде пары ключ/значение.
    Позволяет отделить конфигурацию от образов контейнеров, что делает приложение более переносимым и управляемым
    Могут храниться различные проперти, ямлики, json, xml, переменные окружения и так далее.
    Можно настроить так, что при пересборке образов прокидывались нужные проперти

kubectl create -f pod.yaml
    команда на создание пода в кластере кубера
    вызвав больше одного раза эту команду на манифест с одинаковым name - будет error и под не создастся
    kubectl - тулза (инструмент)
    create - создать
    -f - из файловой системы
    pod.yaml - имя файла манифеста

kubectl apply -f pod.yaml
    команда на создание либо обновление пода (как CREATE or UPDATE).
    Если мы поменяли манифест, то использовать эту команду а не create.

kubectl get pod
kubectl get po
    команда показывает поднятые поды
kubectl get replicaset
kubectl get rs
    команда показывает поднятые репликасеты
kubectl get deployment
kubectl get deploy
    команда показывает поднятые deployment

kubectl delete pod --all
kubectl delete pod my-pod-q73fk
    --all - удалить все поды (лучше не использовать на проде)
    my-pod-q73fk - удалить по имени пода

kubectl scale --replicas=2 replicaset my-replicaset
    Команда на изменение какого либо параметра в манифесте.
    Не меняет сам манифест, а изменяет например количество реплик в рантайме через командную строку.
    Лучше так не делать, так как реальное количество поднятых реплик будет не стыковаться с манифестом
    Могут использовать девопсы чтобы в рантайме что то поменять временно

kubectl describe replicaset my-replicaset
kubectl describe configmap my-configmap
    Команда показывает всю информацию о репликасете или другом объекте k8s

kubectl rollout undo deployment my-deployment
    Откатывает версию манифеста deployment к предыдущей. Например меняем версию - делаем apply и потом эту команду.
    Это можно сделать за счет того, что Deployment хранит старые версии ReplicaSet,
    чтобы можно было к ним вернуться

kubectl apply -f configmap.yaml -f deployment-configmap.yaml
    поднимается конфиг мапа и deployment

kubectl exec -it my-deployment-7fdb7c5f85-j6f4n -- bash
    Команда, чтобы зайти внутрь пода.
    Можно посмотреть структуру файловой системы, конфиги, ввести команды изнутри контейнера и т.д.
